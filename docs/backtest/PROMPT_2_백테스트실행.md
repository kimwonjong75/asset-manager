# [프롬프트 2] 터틀 포트폴리오 백테스트 실행

> 사용법: [프롬프트 1] 완료 후(= `scripts/backtest/data/universe.json` 존재 + AMBIGUOUS 종목 확정 후) 새 Claude Code 세션에 이 파일을 지시.
> 전제 미충족 시 진행하지 말고 프롬프트 1부터 하라고 안내할 것.

---

## 역할과 목표

너는 이 리포지토리에서 **터틀 트레이딩 포트폴리오 백테스트 시뮬레이터를 만들고 실행**한다.
목적은 아래 3가지 결정을 **데이터로** 내리는 것이다:

1. `maxUnitsPerPosition` 기본값: 1 vs 2 vs 4 (= 불타기 강도)
2. 라우팅: 터틀을 위성 후보에만 적용 vs 전 종목 적용 vs 바이앤홀드
3. 코어:위성 비율: 90/10 vs 80/20 vs 70/30

판단 기준은 **승률이 아니라 CAGR·MDD·손익비·연속손실·최악 3년 누적 수익**이다. 승률은 참고 지표로만 표기.

## 배경 (자립 컨텍스트)

- 사용자: 본업 있음, 하루 10분·1~2회 앱 확인. 과거 94종목/60종목 손실 — 문제는 실행·규율. 3년 연속 마이너스를 견디기 어려운 성향.
- 기존 외부 백테스트 참고치(재현 대상 아님, 방향 참고): 불타기(4유닛) CAGR +10.5%/MDD −39.1% vs 불타기 없음 +14.3%/−23.0%. 니케이 터틀 MDD −44%(터틀 부적합 자산의 예).
- 앱 터틀 엔진 파라미터(기본): entry 55일 돌파 / exit 20일 신저가 / 손절 2N / N=ATR20 / 피라미딩 +0.5N 간격 / riskPerUnitPct 0.5%(위성예산 대비) / 총리스크 한도 12% / 1종목 상한 25%.

## 구현 사양

### 시뮬레이터
- 위치: `scripts/backtest/turtlePortfolio.ts` (+ 필요 시 동일 폴더에 모듈 분리). 실행: `npx --yes tsx scripts/backtest/turtlePortfolio.ts --config <이름>`.
- **앱의 순수 함수를 최대한 재사용할 것**: `utils/turtleEngine.ts` (evaluateEntry/evaluatePyramid/evaluateStop/evaluateExit/computeUnitSize), `utils/donchianChannel.ts`. 백테스트가 곧 실제 프로덕션 로직 검증이 되도록. 재사용이 불가능한 부분만 시뮬레이터 내부에 구현하고 그 사유를 주석으로.
- 앱 소스코드는 **읽기만** 하고 수정 금지. 수정 없이 재사용이 안 되면(예: export 누락) 최소 export 추가만 허용하되 변경 목록에 명시.

### 데이터
- `scripts/backtest/data/universe.json` 의 `dataSymbol`/`proxySymbol` 로 일봉 OHLC fetch (Yahoo chart API).
- 캐시: `scripts/backtest/data/cache/{symbol}.json` — 재실행 시 재다운로드 금지. (`data/` 는 gitignore 상태 유지)
- 환율: `KRW=X`, `JPYKRW=X` 일봉으로 히스토리컬 환산. 환율 데이터 없는 날은 직전값 carry-forward.
- 프록시 사용 종목은 리포트에 "프록시 기반" 명시.
- 일부 종목 fetch 실패 시 해당 종목만 제외하고 계속 진행(부분 성공), 제외 목록 보고.

### 체결 규칙 (일봉 근사 — 반드시 이대로, 룩어헤드 0)
- 채널·N 계산: **당일 제외**(excludeToday), 신호 판정은 당일 고가/저가 vs 전일까지의 채널.
- 진입: `high[t] > donchianHigh(55, t-1까지)` → 체결가 = `max(open[t], donchianHigh)` (갭상승이면 시가 체결).
- 청산: `low[t] < donchianLow(20, t-1까지)` → 체결가 = `min(open[t], donchianLow)`.
- 손절: `low[t] <= stopPrice` → 체결가 = `min(open[t], stopPrice)`. 청산·손절이 같은 날이면 손절 우선.
- 피라미딩: `high[t] >= 마지막체결가 + 0.5N` → 체결가 = `max(open[t], trigger)`. 진입과 동일 봉에서 연쇄 피라미딩 금지(1일 1유닛).
- 비용: 왕복 0.2% (편도 0.1%씩 체결가에 반영).
- 신고가/신저가 판정은 원통화, 사이징·평가액 집계는 당일 환율로 KRW 환산.

### 포트폴리오 규칙
- 코어: universe의 CORE 종목을 현재 비중 비례(또는 동일가중 — 둘 다 돌려 민감도 확인)로 보유, **반기 1회(1월/7월 첫 거래일) 리밸런싱**, 비용 동일 적용.
- 위성: SATELLITE_TURTLE 종목에 터틀 규칙 적용. 위성예산 = 총자산 × 위성비율. 미투자분은 현금(무이자 가정, 한계로 명시).
- 시작 자본 1억 KRW (상대 성과만 보므로 임의값).

## 실험 매트릭스

공통 기간: 데이터가 허용하는 최장 구간(가능하면 2015-01-01~최신). **2022 하락장 반드시 포함.** 공통 시작일은 "유니버스 내 최소 필수 종목의 교집합"으로 잡되, 늦게 상장한 종목은 프록시로 보완.

| # | 실험 | 변수 | 고정 |
|---|---|---|---|
| A | 불타기/유닛 | maxUnits 1 / 2 / 4 | 위성 종목만, 55/20, risk 0.5% |
| B | 라우팅 | (i) 위성만 터틀 (ii) 전 종목(코어 포함) 터틀 (iii) 전 종목 B&H (iv) 코어 B&H+위성 터틀 | A의 최적 유닛 |
| C | 코어:위성 비율 | 90/10 / 80/20 / 70/30 | B의 최적 라우팅 |
| D | 리스크 보정 | riskPerUnitPct 0.5 / 1.0 | A에서 이긴 유닛 위에서. 유닛이 낮게 확정돼 위성 노출이 과소하면 이 손잡이로 보정 |
| E (선택) | 진입 채널 | 55 / 100 / 200일 | 여력 있으면 |
| F (선택) | 청산 | 20일 신저가 vs 샹들리에(최고가−3ATR) | 여력 있으면 |
| G (선택) | 상관그룹 한도 | 그룹당 리스크 한도 ON/OFF (귀금속/암호화폐/원자재 그룹) | 2024 동반 휩소 구간 주목 |

## 평가 지표 (config별 전부)

- CAGR, MDD, Calmar(CAGR/|MDD|), 손익비(profit factor)
- 거래 횟수(사용자 부담 — 월평균 액션 수로도 환산), 승률(참고)
- 최대 연속 손실 거래 수, **최악 3년 롤링 누적 수익률** (사용자가 견딜 수 있는가의 지표)
- 연도별 수익률 표
- 위성 파트 단독 지표 + 코어 합산 포트폴리오 지표 분리 표기

## 과적합 방지

- E/F/G 같은 파라미터 선택 실험은 **train 구간(시작~2023-12)** 에서 고르고, **holdout(2024-01~최신)** 에서 재확인. holdout에서 뒤집히면 "과적합 의심"으로 보고하고 채택하지 말 것.
- A/B/C는 구조 비교이므로 전 구간 결과 + 구간별(상승장/2022 하락장/2024~) 분해를 함께 표기.
- **유닛 수(A)는 사전에 정답을 두지 말 것.** 외부 참고치(4유닛이 나빴다)는 방향 참고일 뿐, 이 유니버스·이 기간에서 1/2/4를 실제로 돌려 CAGR·MDD·최악 3년 누적으로 판정한다. 특정 유닛을 전제로 다른 실험을 진행하지 말 것.
- 지표가 좋아도 거래 횟수가 사용자 리듬(하루 1~2회 확인)을 초과하면 감점 사유로 명시.

## 제약·금지

- 앱 런타임 동작 변경 금지 (시뮬레이터는 독립 스크립트).
- 백테스트 결과로 `DEFAULT_TURTLE_SETTINGS` 를 **자동 변경하지 말 것** — 권고만 하고 변경은 사용자 승인 후 별도 작업.
- 결과 수치를 낙관 편향으로 요약하지 말 것: 나쁜 결과도 표에 그대로. "모든 config가 B&H에 진다" 같은 결론도 유효한 결론이다.
- 커밋/푸시 금지 (사용자가 GitHub Desktop으로 직접).
