## 0. 2판 개정 사유 — 1판의 전제가 틀렸습니다

1판은 **"엔진은 멀쩡하니 config 한 줄만 추가하면 20/10이 오늘 돌아간다"**를 전제로 세워졌습니다. **이 전제가 틀렸습니다.**

엔진은 lookback을 정확히 전달합니다. 그러나 **그 lookback이 먹는 데이터가 오염돼 있습니다.** 지금 config만 추가해 돌리면 오염된 숫자 위에 새 오염을 얹을 뿐입니다.

1판에서 틀렸던 주장(실측으로 반증됨):

| 1판 주장 | 실측 결과 |
|---|---|
| 주말 carry-forward로 TR=0 → N 30% 과소 → 수량 45% 과대 | **틀림.** TR은 0이 아니라 **직전 거래일 레인지가 반복**됨. ATR 수준 차이 **−0.8%** |
| N 왜곡이 계좌 위험을 키움 | **정반대.** 수량↑와 손절거리↓가 **정확히 상쇄** → 손절 손실 **불변(−0.0%)**. 커지는 건 **명목노출(+43%)** |
| 12일 데이터로 잘못된 주문이 나감 | **불가능.** ATR이 21일 미만이면 `null` → 진입 거절. 문제 구간은 **21~54일** |

**이 오류들은 코드를 읽고 추론해서 나온 것입니다.** 그래서 이 계획에는 아래 규칙을 넣습니다.

> ### 📏 측정 규칙 (필수)
> **모든 왜곡·편향 주장은 합성 또는 실데이터로 측정한 수치를 첨부한 뒤에만 판단 근거로 사용한다. 코드 독해에 근거한 크기 추정은 리포트에 쓰지 않는다.**
> 이 문서의 모든 수치는 실행해서 얻은 것이며, 출처를 함께 표기한다.

## 1. 왜 하는가

### 55/20은 미검증입니다

`docs/backtest/REPORT_터틀백테스트.md:100`:
> "E(진입 채널 55/100/200)… 는 스펙상 선택 사항으로 **이번 실행에서는 생략했다.**"

실행된 실험(A~D, 강건성 R1~R3)은 **전부 55/20을 고정**하고 유닛 수·표본만 바꿨습니다. 그럼에도 `docs/운용설정_따라하기_매뉴얼.md:38`은 **"백테스트가 확정한 값 그대로다"**라고 기술합니다 — **진입 채널에 대해서는 사실이 아닙니다.**

### 더 나쁜 사실: 기존 백테스트는 55/20을 테스트한 적이 아예 없습니다

합집합 캘린더가 거래일의 **1.40배** 행을 만들기 때문에(실측), 백테스트의 "55행"은 55거래일이 아닙니다.

| 백테스트가 부르는 이름 | **실제 (실측)** |
|---|---|
| 진입 돈치안 **55일** | **39.4 거래일** |
| 청산 돈치안 **20일** | **14.3 거래일** |
| ATR **20일** | **14.3 거래일** |

**즉 기존 리포트는 55/20이 아니라 약 39/14 시스템의 성과입니다.** 앱이 실제로 돌리는 전략과 다른 전략을 측정한 것입니다.

→ **따라서 4단계(현재 설정 재실행)는 "기존 대비 변화량 확인"이 아니라 앱 실제 설정에 대한 _최초 측정_입니다.** 기존 리포트와 숫자가 크게 달라도 회귀가 아니라 정상입니다. 이 구분을 안 해두면 나중에 엉뚱한 곳을 디버깅하게 됩니다.

**참고:** 지금 그대로 20/10을 돌리면 실제로는 **14/7**이 됩니다. 캘린더를 고치기 전에는 사용자가 궁금해한 질문에 답할 수 없습니다.

## 2. 판정 — 기존 리포트 결론은 전부 잠정입니다

유닛 2, 위성비중, 성과 권고 **모두 재검증 전까지 잠정**입니다. 특히:

### 실험 D(riskPerUnitPct 0.5 vs 1.0)는 상당 부분 측정 자체가 성립하지 않았습니다 (신규 발견)

`positionValueCapPct=25`가 **ATR/가격 ≤ 2%인 자산에서 두 설정을 같은 수량으로 만듭니다.** 실측:

| ATR/가격 | 수량(0.5%) | 수량(1.0%) | |
|---|---|---|---|
| 0.8% | 250,000 | 250,000 | ❌ 동일 — 실험 무효 |
| 1.2% | 250,000 | 250,000 | ❌ 동일 — 실험 무효 |
| 2.0% | 250,000 | 250,000 | ❌ 동일 — 실험 무효 |
| 3.0% | 166,666 | 250,000 | ✅ 다름 |
| 5.0% | 100,000 | 200,000 | ✅ 다름 |

즉 실험 D는 **ATR/가격 > 2%인 자산 — 사실상 코인 — 만 구분했습니다.** 기존 리포트에 "44거래가 전부 BTC였던" 이력이 있음을 감안하면, D의 결론은 거의 코인 한 종목이 만들었을 가능성이 큽니다.

**함의:** 상한을 고정한 채 `riskPerUnitPct`만 스윕하면 저변동 자산에서는 영원히 무효 실험입니다. **리스크 파라미터를 다시 고를 거라면 `positionValueCapPct`와의 상호작용을 반드시 함께 설계해야 합니다.**

## 4. 실행 순서 (확정)

### 1단계 — 종목별 실제 거래일로 채널·ATR 계산
공통 캘린더는 **평가액 집계에만** 사용합니다. 채널과 ATR은 각 종목의 실제 거래일 시계열에서 계산합니다. (문제 1·2 해소)

### 2단계 — 데이터 정합성
- 프록시의 **실제 호가통화**와 **일별 KRW 환율** 적용 (문제 3)
- 분할조정 OHLC, 배당 포함 비교수익
- 데이터 공급자·수집일 기록
- 수수료·세금·환전·슬리피지 기본값 + **2배 스트레스값**

### 3단계 — 앱과 같은 체결 모델
**주평가 = `종가 확인 → 다음 거래일 시가 체결`** (앱의 관찰 방식 근사).
장중 돌파 체결은 **참고 결과로 분리**합니다. (문제 4)

### 4단계 — 현재 설정 최초 측정
수정된 데이터로 **`55/20 · 최대 2유닛 · 리스크 0.5% · 상한 25%`**를 그대로 실행.
※ 기존 리포트와의 차이는 **회귀가 아니라 최초 측정 결과**입니다(§1).

### 5단계 — 채널만 변경해 신호 효과 분리
같은 조건에서 **채널만 `20/10`**으로 변경. 다른 모든 변수 고정.

### 6단계 — 안정성 검사 (최적점 찾기 아님)
- 빠른 계열: 진입 `15·20·25` × 청산 `5·10·15`
- 느린 계열: 진입 `40·55·70` × 청산 `15·20·30`
- 필요 시에만: `100·150·200`

**가장 좋은 한 점을 고르지 않습니다.** 인접 값에서도 성과가 유지되는 **넓은 안정 구간**을 찾습니다. 시험했거나 버린 모든 조합을 기록하고 **과최적화 확률(PBO)**을 평가합니다. [Probability of Backtest Overfitting](https://doi.org/10.21314/JCF.2016.322)

### 7단계 — 유닛 수·위성비중 재선택 (조건부)
다시 고른다면 **채널과 분리된 train fold에서만** 수행하고, **최종 lockbox는 단 한 번만** 평가합니다. (문제 5 해소)
**⚠️ §2에 따라 `positionValueCapPct`와의 상호작용을 함께 설계할 것.** 상한 고정 상태로 리스크만 스윕하면 저변동 자산에서 무효 실험이 반복됩니다.

### 8단계 — 검증 종목군 분리
| 종목군 | 용도 |
|---|---|
| **앱 재현군** — 현재 터틀 후보 관심종목 + 열린 포지션 | 앱 동작 재현 |
| **개인 적용성군** — 현재 보유 / 과거 매도 / 관심종목을 구분 | 개인화 확인 |
| **자산군 대표군** — 한국·미국·글로벌 주식, 국채, 금·원자재, 리츠, 환율, 코인 | 전략 유효성 |
| **일반화 검증군** — point-in-time 자료 (있을 경우만) | 일반화 |

장기 방향성은 **지수·선물 자료**로, 실제 투자 가능성은 **ETF의 실제 상장 이후 자료**로 각각 평가합니다.

### 9단계 — 생존편향 처리
**현재 보유종목은 이미 살아남아 보유 중인 종목입니다.** 개인 적용성 확인엔 유용하지만 **전략 자체의 유효성 근거로 쓰면 안 됩니다.**
상장폐지 포함 자료가 없으면 **생존편향을 명시하고 개별주 일반화 결론을 내리지 않습니다.**
유료 point-in-time 자료(CRSP·Norgate 등)는 **선택 검증 단계로 분리**합니다 — 현재 데이터 소스(자체 Cloud Run `/history`)에는 상장폐지 종목의 당시 시점 자료가 없어 이번 범위에 넣지 않습니다.

## 6. 평가 기준

"안정적인 수익 극대화"는 CAGR 하나로 판정하지 않습니다.

- **비용 차감 OOS CAGR**
- **최대낙폭(MDD)과 회복기간**
- **Calmar**
- **최악 rolling 1년 · 3년**
- **거래당 R-multiple** (기대수익 ÷ 초기 위험)
- **거래 수, 회전율, 최대 연속 손실**
- **한 종목·한 자산군의 이익 기여 집중도** ← 코인 한 종목이 결과를 만드는지 확인
- **스트레스**: 비용 2배, 하루 지연 체결, 갭 악화
- **비교 대상**: Buy & Hold, 현금 대기

### 최종 선택은 파레토 기준

- 비슷한 CAGR에서 낙폭이 작거나
- 비슷한 낙폭에서 CAGR이 높고
- **미사용 기간과 인접 파라미터에서 유지되며**
- **특정 종목이나 코인 한두 개가 결과를 만들지 않을 것**

### 채택 문턱

> **`20/10`은 같은 비용·체결 조건에서 `55/20`보다 OOS Calmar 또는 회복기간을 개선하고, 인접 파라미터와 비용 2배 조건에서도 유지될 때만 채택합니다.**

55/20은 원전에서 온 값이므로 **바꾸려면 입증 책임이 도전자 쪽에 있습니다.** 명확히 지지 않으면 유지합니다.

**앱 기본값(`DEFAULT_TURTLE_SETTINGS`)은 이 실험으로 자동 변경하지 않습니다.** 권고만 하고 사용자가 별도 승인합니다 (`PROMPT_2:92-97`).

## 8. 한계 (리포트에 반드시 명시)

- **생존편향** — 유니버스가 오늘의 보유 종목을 과거로 역투영. 망한 종목 없음. **이번엔 없애지 못하므로 명시하고 결론 강도를 낮춤.**
- **가중치 룩어헤드** — `weightPct`가 현재 시가 비중인데 과거 목표 비중으로 쓰임. equal-weight 민감도로만 완화.
- **프록시 종목 14개** — 방향성은 유효하나 절대 수익률은 벌어짐.
- **AMBIGUOUS 19종목 제외** — 분류 확정 시 재실행 필요.
- **현금 0% · 세금/환전 스프레드 미반영** (2단계에서 개선 대상).
- **`drawdownScalingEnabled: false`** — 단 이 설정은 현재 **앱에서도 무효**임(`turtleEngine.ts:292` 호출처 없음). 별도 이슈.
- **사이징 예산은 반기 리밸런싱 때만 갱신** — 일일 갱신 시 재귀적 폭주(BTC 1거래가 수백조 원) 발생. **절대 "고치지" 말 것.**
- **`all-turtle`·`all-bh` 라우팅은 구조적 결함으로 판정 제외.** 모든 실험은 `two-bucket`으로 실행.
- **ATR 워밍업 버그** — `calculateATR`은 워밍업 창 안에 null이 하나라도 있으면 **영구 null** 반환. 시뮬레이터는 `firstValidIdx` 슬라이싱으로 우회 중. **새 코드 경로는 반드시 이를 재사용할 것** (미준수 시 "BTC만 거래되는" 과거 버그 재현).

## 9. 부록 — 이 문서 수치의 출처

전부 스크래치패드에서 실제 실행해 얻은 값입니다(저장소에 커밋하지 않음).

| 수치 | 방법 |
|---|---|
| 주말 TR = 직전 레인지(0 아님) | 합성 주식(월~금) → `alignToCalendar` → TR 직접 계산 |
| ATR 수준 차이 −0.8% | 변동 레인지 합성 250거래일, `calculateATR` 거래일 vs 합집합 |
| 55행=39.4거래일, 20행=14.3거래일 | 합집합/거래일 행 비율 1.40배로 나눔 |
| 진입 가능 최소 21일 | `computeN`+`calculateDonchianHigh`를 12·20·21·25·40·54·55·80일로 반복 |
| 리스크 0.5% → 실손실 1.00% | `computeUnitSize`(위성예산 1억, N=2, 가격 100, 상한 0) |
| N −30% → 손실 불변, 명목 +43% | 같은 함수, N=2 vs 1.4 |
| 상한은 ATR/가격 < 2%에서 걸림 | `computeUnitSize` + `DEFAULT_TURTLE_SETTINGS`, ATR/가격 0.8~8% |
| 실험 D 무효 구간 | 같은 조건에서 `riskPerUnitPct` 0.5 vs 1.0 수량 비교 |

**재현 시 주의:** `computeUnitSize`의 인자 순서는 `(settings, n, price, opts)`입니다. `(settings, price, n, ...)`으로 넣으면 조용히 틀린 수량이 나옵니다(초안 작성 중 실제로 겪음).
